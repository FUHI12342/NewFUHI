# Timebaibai SaaS化 草案

> 作成日: 2026-03-14 | ステータス: **草案（将来実装予定）**
> 対象: timebaibai.com をマルチテナントSaaSとして商用化

---

## 1. ビジョン

現在の NewFUHI（1店舗専用デプロイ）を、オンライン申込 → 審査承認 → 個別環境自動生成の
**セルフサービスSaaS** に発展させる。

```
[顧客] → timebaibai.com 申込フォーム
          ↓
[管理者] → 審査・承認
          ↓
[自動プロビジョニング] → AWS上にテナント環境生成
          ↓
[顧客] → https://<store-slug>.timebaibai.com でログイン・利用開始
```

## 2. アーキテクチャ選定

### 2-A. テナント分離方式の比較

| 方式 | メリット | デメリット | コスト/テナント |
|------|---------|-----------|----------------|
| **A: 完全分離** (EC2+RDS per tenant) | データ完全分離、カスタマイズ自由 | コスト高、運用複雑 | ¥5,000〜9,500/月 |
| **B: 共有インフラ+DB分離** (1 EC2 + DB per tenant) | バランス良し | アプリ共有のためバージョン管理要注意 | ¥3,000〜5,000/月 |
| **C: 完全共有** (マルチテナントDB) | 最低コスト | Django改修大、データ漏洩リスク | ¥500〜1,500/月 |

### 推奨: **方式B（共有インフラ+DB分離）** → 将来的に方式Aへ移行可能

理由:
- Django の `DATABASE_ROUTERS` + `DATABASES` 動的設定で実現可能
- テナント間のデータ分離が確保される
- 1台の EC2 (t3.medium) で10〜20テナントを収容可能
- RDS は db.t3.micro (¥2,847/月) をテナント毎に立てるか、1つの RDS に複数 DB を作成

### 段階的移行パス

```
Phase 1: 方式C（プロトタイプ）
  - django-tenants ライブラリで PostgreSQL スキーマ分離
  - 1つの EC2 + 1つの RDS で全テナント
  - 月額インフラ: ~¥7,000（固定）

Phase 2: 方式B（本番）
  - テナント毎に RDS データベース or スキーマ
  - ECS Fargate でコンテナ化
  - ALB + Route53 でサブドメインルーティング

Phase 3: 方式A（大口顧客向け）
  - 専用EC2 + 専用RDS
  - Terraform/CDK で自動プロビジョニング
```

## 8. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| テナント間データ漏洩 | 致命的 | django-tenants のスキーマ分離 + 徹底テスト |
| 1テナントの過負荷が全体に影響 | 高 | リソース制限 (cgroups) + 監視 + 大口は専用インフラ |
| IoTデバイス故障率 | 中 | 予備在庫確保、リモート診断機能 |
| Stripe/決済トラブル | 中 | Webhook冪等性、手動復旧フロー |
| 法務 (利用規約/SLA違反) | 中 | 弁護士レビュー、SLA明文化 |
| サポート負荷増大 | 中 | FAQ充実、チャットボット、ティア別対応 |

## 9. 競合との差別化ポイント

1. **オールインワン**: POS + 予約 + シフト + 勤怠 + 給与 + 分析 + IoT を1サービスで提供
   - 競合は単機能サービスを組み合わせる必要あり (¥32,000+)
2. **IoTセンサー統合**: CO/温湿度/人感の自動モニタリング + 換気自動制御
   - 競合に同等のサービスなし（日本の飲食向けSaaSでは唯一）
3. **高度分析**: RFM, コホート, バスケット分析, 売上予測, CLV
   - Square Analytics や Toast に匹敵する分析を中小飲食店に提供
4. **価格優位性**: 個別導入比 50〜60% の価格
5. **シーシャ/バー特化**: ガスセンサー対応は業界唯一
