1. システム概要

1.1 NewFUHI とは

NewFUHI は、占いサロン・飲食店・宿泊施設などの店舗運営を統合的に管理する Django ベースの Web プラットフォームです。予約管理、決済、IoT デバイス制御、在庫管理、シフト・給与管理、EC ショップ、物件監視など幅広い業務をカバーします。

1.2 技術スタック

レイヤー技術
バックエンドDjango / Django REST Framework / Python 3
タスクキューCelery + Redis
データベースSQLite（開発）/ PostgreSQL・RDS（本番）
認証Django Auth + LINE OAuth2（python-social-auth）
暗号化cryptography（Fernet）
QRコードqrcode[pil]
AWS連携boto3
Web サーバーNginx + Gunicorn
CI/CDGitHub Actions
プロセス管理systemd

7. 予約フロー

7.1 LINE 予約フロー

  1. 顧客がトップページで店舗 → スタッフ → 日時を選択
  2. /booking/channel-choice/ で LINE を選択
  3. /line_enter/ → LINE OAuth2 認証開始
  4. LINE ログイン成功 → /booking/login/line/success/ コールバック
  5. 仮予約（Schedule）作成（is_temporary=True）
  6. LINE User ID を暗号化（Fernet）＋ハッシュ（SHA-256 + pepper）して保存
  7. 決済 URL（Coiney）を生成し payment_url に保存
  8. 顧客が決済完了 → Webhook で is_temporary=False に更新
  9. チェックイン QR コード生成・保存
仮予約は Celery タスク（毎分実行）で 10分超過分を自動削除します。

7.2 メール予約フロー

  1. 顧客が /booking/channel-choice/ で「メール」を選択
  2. /booking/email/ でメールアドレス入力
  3. OTP（ワンタイムパスワード）をメール送信
  4. /booking/email/verify/ で OTP 入力・認証
  5. 認証成功 → 仮予約作成（booking_channel='email', email_verified=True）
  6. 以降は LINE フローと同じ（決済→確定→QR生成）

7.3 チェックイン

  1. 店舗スタッフが /checkin/ で QR スキャナーを起動
  2. 顧客のチェックイン QR コードを読み取り
  3. /api/checkin/ API でチェックイン処理
  4. is_checked_in=True, checked_in_at 記録

7.4 予約キャンセル

/cancel_reservation/<schedule_id>/ で予約をキャンセル。is_cancelled=True に更新。

8. 決済連携

8.1 Coiney 決済フロー

  1. 仮予約作成時に決済 API（PAYMENT_API_URL）に決済リクエスト送信
  2. PAYMENT_API_KEY で API 認証
  3. 決済 URL を取得 → Schedule の payment_url に保存
  4. 顧客を決済ページにリダイレクト
  5. 決済完了 → Coiney が Webhook を送信

8.2 Webhook 処理

# URL: /api/coiney/webhook/<orderId>/
# ビュー: coiney_webhook(request, orderId)
# 処理:
#   1. orderId で Schedule を検索
#   2. is_temporary = False に更新（確定）
#   3. チェックイン QR コード生成
#   4. LINE で確認メッセージ送信

8.3 PaymentMethod モデル

店舗ごとに複数の決済方法を設定可能：cash / coiney / paypay / ic / other。api_key, api_secret, extra_config（JSON）で各決済プロバイダーの設定を管理。

9. QRテーブル注文

9.1 概要

テーブルに設置した QR コードをスマートフォンで読み取り、メニュー閲覧→カート→注文→会計をセルフサービスで完結するシステム。

9.2 フロー

  1. QR 読み取り → /table/<uuid>/ テーブルメニュー表示
  2. 商品選択 → /api/table/<uuid>/cart/add/ でカート追加
  3. カート確認 → /table/<uuid>/cart/
  4. 注文確定 → /api/table/<uuid>/order/create/
  5. 注文状況 → /api/table/<uuid>/orders/status/ でリアルタイム確認
  6. 注文履歴 → /table/<uuid>/history/
  7. 会計 → /table/<uuid>/checkout/

ORDERED → PREPARING → SERVED → CLOSED
  注文済     調理中      提供済    会計済

スタッフは /api/staff/orders/items/<id>/status/ で各アイテムのステータスを更新。

10. IoTデバイス管理

10.1 デバイス登録

管理画面の「IoT管理」から IoTDevice を登録。デバイスタイプ：multi（マルチセンサー）/ door（ドアセンサー）/ other。

10.2 API 認証

デバイスは X-API-Key ヘッダーで認証。API キーは SHA-256 ハッシュで保存（平文は保存しない）。登録時にのみ平文表示。

10.3 センサーデータ送信

POST /api/iot/events/
Headers: X-API-Key: <api_key>
Body: {
  "event_type": "sensor_reading",
  "mq9_value": 150.5,    // ガスセンサー値
  "light_value": 450.0,  // 照度値
  "sound_value": 35.2,   // 音量値
  "pir_triggered": false  // 人感センサー
}

10.4 ガスアラート

mq9_value が mq9_threshold を超えた場合、Celery タスク trigger_gas_alert が発火。メール＋LINE で緊急通知。

10.5 IR スマートハブ

  1. 管理画面で IRCode（学習済み赤外線コード）を登録
  2. POST /api/iot/ir/send/ でコマンド送信
  3. デバイスが GET /api/iot/config/ で pending_ir_command を取得して実行

10.6 Wi-Fi パスワード管理

デバイスの Wi-Fi パスワードは Fernet 暗号化して保存。set_wifi_password() / get_wifi_password() メソッドで暗号化・復号。

10.7 センサーダッシュボード

/dashboard/sensors/ でリアルタイムセンサーデータを可視化。API: /api/iot/sensors/data/, /api/iot/sensors/pir-events/, /api/iot/sensors/pir-status/。

12. シフト・勤怠・給与管理

12.1 シフト管理フロー

  1. シフト期間作成：管理画面で ShiftPeriod 作成（対象月・締切設定、status=open）
  2. シフト希望提出：スタッフが /shift/<period_id>/submit/ で希望提出（available / preferred / unavailable）
  3. シフト確定：管理者が ShiftAssignment を作成（日付・開始時刻・終了時刻）

12.2 勤怠管理

12.3 給与計算

  1. PayrollPeriod 作成（status=draft）

  5. PayrollDeduction で社会保険料・税金を自動控除

15.2 セキュリティ監査

python manage.py security_audit または Celery タスク（毎日03:00）で自動実行。チェックカテゴリ：

  • django_settings — DEBUG, ALLOWED_HOSTS, SECRET_KEY 等
  • credentials — API キー・パスワードの安全性
  • endpoints — エンドポイントのアクセス制御
  • infrastructure — SSL, ファイアウォール
  • dependencies — パッケージの脆弱性

15.3 監視ログ（SecurityLog）

SecurityAuditMiddleware が全リクエストを監視し、以下のイベントを自動記録：

event_typeseverityトリガー
login_successinfoログイン成功
login_failwarningログイン失敗
api_auth_failwarning/api/ への 401/403
permission_deniedwarning403 レスポンス
suspicious_requestcriticalレート制限超過
admin_actioninfo管理画面操作

15.4 レート制限

同一 IP から60秒間に100リクエスト超で suspicious_request（critical）を記録。カウンターは最大10,000エントリ。IP 検出は X-Forwarded-For ヘッダー対応。

15.5 本番セキュリティ設定

  • SECURE_SSL_REDIRECT = True
  • SECURE_HSTS_SECONDS = 31536000（1年）
  • SESSION_COOKIE_SECURE = True
  • CSRF_COOKIE_SECURE = True
  • X_FRAME_OPTIONS = "DENY"

15.6 ログ保持

cleanup_security_logs タスク（毎週日曜04:00）で90日以上前のログを自動削除。--days オプションで保持期間変更可能。
